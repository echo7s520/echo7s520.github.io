一、 Nginx 简介

1、Nginx 是高性能的 Http 和反向代理的 服务器 ，因它的稳定性、丰富的功能集、简单的配置文件和低系统资源的消耗而闻名。处理高并发能力是十分强大，能经受高负载的考验，能够支持高达 50000 个并发连接数的响应。

2、特点

  • 正向代理
  • 反向代理
  • 负载均衡
  • 动静分离
  • 高可用集群

第三部分 : http 块

Nginx 服务器配置最频繁的内容，代理，缓存，日志定义等大多数功能和第三方模块的配置都在这里。（http 块 包括 http 块 + server 块）

http 块 : http 全局块配置的指令包括文件引入、 MIME-TYPE 定义、日志自定义、连接超时时间、单链接请求数上限等。

server 块 : 这块和虚拟主机有密切关系，虚拟主机从用户角度看，和单一独立的硬件主机是完全一样的，该技术的产生是为了节省互联网服务器硬件成本。（server 块 包括 全局 server 块 + location 块）

​ 全局 server 块 : 最常见的配置是本虚拟机主机的监听配置和本虚拟主机的名称或 IP 配置。

​ location 块 : 这块的主要作用是基于 Nginx 服务器接收到的请求字符串，对虚拟主机名称之外的字符串进行匹配，对特定的请求进行处理。地址定向、数据缓存和应答控制等功能。

四、正向代理

正向代理 : 需要在客户端配置代理服务器进行指定网站访问。（例: 挂 VPN 访问外网）

（ 正向代理 : 隐藏客户端 ）

五、反向代理

反向代理 : 客户端对代理是无感知，因为不需要客户端进行配置，我们只需要将请求发送到反向代理服务器，由反向代理服务器进行选择获取目标服务器数据，进行返回给客户端。此时，反向代理服务器和目标服务器对外就是一个服务器，暴露的是代理服务器，隐藏了真实服务器 ip 地址。

（ 反向代理 : 隐藏服务端）

Nginx 进行请求转发配置

location 指令说明

  1. = ：用于不含正则表达式的 uri 前，要求请求字符串与 uri 严格匹配，如果匹配成功，就停止继续向下搜索并立即处理该请求。
  2. ~：用于表示 uri 包含正则表达式，并且区分大小写。
  3. ~*：用于表示 uri 包含正则表达式，并且不区分大小写。
  4. ^~：用于不含正则表达式的 uri 前，要求 Nginx 服务器找到标识 uri 和请求字符串匹配度最高的 location 后，立即使用此 location 处理请求，而不再使用 location 块中的正则 uri 和请求字符串做匹配。

六、负载均衡

负载均衡 : 客户端发送多个请求服务器时，服务器请求处理，与数据进行交互，返回给客户端。当请求很多，服务器访问数据较大时，可以采用 升级服务器配置 。但 当数据非常大时，服务器配置升级有限，这时，采用 负载均衡 方法进行解决。

即 : 增加服务器数量，将原先请求集中到一个服务器的情况改为请求分发到多个服务器上，将负载分发不同服务器上，实现请求响应。

例：当 一匹马 拉车时，货物比较重，可以选择换一匹强壮的马，但是，货物非常重，马的强壮也是有限的。这时，可以多叫几匹马，一起拉更快，更轻松。

实现效果 : 浏览器地址栏输入地址 http://127.0.0.1/helloworld，负载均衡效果，平均 8081 和 8082 端口中。

负载均衡分配策略

​ 轮询

Nginx 默认的分配策略，每个请求按时间顺序逐一分配到不同的后端服务器，如果服务器宕机，自动删除分配。

​ 加权

weight 代表 权，默认为 1，权重越高，被分配的客户端越多，指定轮询几率，weight 和访问比率成正比，用于后端服务器性能不均匀的情况。

​ ip_hash

每个请求按访问 ip 的 hash 结果分配，这样每个访客固定访问一个后端服务器，可以解决 session 的问题。

​ fair

这是 Ngnix 负载均衡第三方分配策略。按后端服务器的响应时间来分配请求，响应时间短的优先分配。

七、动静分离

动静分离 : 为了加快网站的解析速度，可以把动态页面和静态页面由不同服务器来解析，加快解析速度。降低原来单个服务器压力。

​ 实现方法1 : 把纯粹的静态文件独立成单独的域名，放在独立的服务器上，也是目前主流的方案

​ 实现方法2 : 把动态和静态文件放一起发布，通过 Nginx 分开（不适合经常变动的资源）

实现效果 :

  • 浏览器地址栏输入地址 http://127.0.0.1/image/1.png，显示 图片
  • 浏览器地址栏输入地址 http://127.0.0.1/helloworld，显示 Tomcat

九、Nginx 原理

master-workers 的机制的好处

  • 对于每个 worker 进程来说，独立的进程，不需要加锁，所以省掉了锁带来的开销，同时在编程以及问题查找时，也会方便很多。

  • 采用独立的进程，可以让互相之间不会影响，一个进程退出后，其它进程还在工作，服务不会中断， master 进程则很快启动新的 worker 进程。

  • worker 进程的异常退出，肯定是程序有 bug 了，异常退出，会导致当前 worker 上的所有请求失败，不过不会影响到所有请求，所以降低了风险。

需要设置多少个 worker

Nginx 同 redis 类似都采用了 io 多路复用机制，每个 worker 都是一个独立的进程，但每个进程里只有一个主线程，通过异步非阻塞的方式来处理请求， 即使是千上万个请求也不在话下。每个 worker 的线程可以把一个 cpu 的性能发挥到极致。

所以 worker 数和服务器的 cpu 数相等是最为适宜的。设少了会浪费 cpu，设多了会造成 cpu 频繁切换上下文带来的损耗。

连接数 worker_connection

这个值是表示每个 worker 进程所能建立连接的最大值，所以，一个 nginx 能建立的最大连接数，应该是 worker_connections * worker_processes。当然，这里说的是最大连接数，对于 HTTP 请求本地资源来说，能够支持的最大并发数量是 worker_connections * worker_processes

如果是支持 http1.1 的浏览器每次访问要占两个连接，所以普通的静态访问最大并发数是：worker_connections * worker_processes /2，而如果是 HTTP 作 为反向代理来说，最大并发数量应该是 worker_connections * worker_processes/4。因为作为反向代理服务器，每个并发会建立与客户端的连接和与后端服务的连接，会占用两个连接。
